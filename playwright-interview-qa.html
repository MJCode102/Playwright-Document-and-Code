FRAMEWORK

Framework & Design

Q1. How do you structure your Playwright automation framework? ▶

A well-structured Playwright framework follows the Page Object Model (POM) pattern with separation of concerns:

project/
├── pages/                  # POM classes — locators + actions
│   ├── base.page.js        # common: navigate, waitFor, screenshot
│   ├── login.page.js
│   └── dashboard.page.js
├── tests/                  # spec files only — assertions here
│   ├── login.spec.js
│   └── dashboard.spec.js
├── utils/                  # reusable helpers
│   ├── auth-helper.js      # OAuth / SSO login
│   ├── test-data-manager.js
│   └── step-logger.js
├── test-data/              # JSON fixtures
│   └── users.json
├── settings/               # env config, setup/teardown
│   ├── env.js
│   ├── global-setup.js
│   └── global-teardown.js
├── auth/                   # saved storageState files
│   ├── admin.json
│   └── viewer.json
├── playwright.config.js
└── .env
Key principles: Pages own locators. Tests own assertions. Utils handle cross-cutting concerns. Config drives environments.
Q2. How do you handle single sign-in across multiple test cases? ▶

Use storageState — log in once in globalSetup, save the auth context, and reuse it in every test.

// settings/global-setup.js
import { chromium } from '@playwright/test';

export default async function globalSetup() {
  const browser = await chromium.launch();
  const page = await browser.newPage();
  await page.goto('https://app.example.com/login');
  await page.fill('#email', 'user@example.com');
  await page.fill('#password', 'Secret1!');
  await page.click('button[type=submit]');
  await page.waitForURL('**/dashboard');
  await page.context().storageState({ path: 'auth/user.json' });
  await browser.close();
}
// playwright.config.js
export default {
  globalSetup: './settings/global-setup.js',
  use: {
    storageState: 'auth/user.json',  // ALL tests reuse this auth
  },
};
// tests/dashboard.spec.js — no login needed!
test('dashboard loads', async ({ page }) => {
  await page.goto('/dashboard');
  await expect(page.locator('h1')).toHaveText('Welcome');
});
storageState saves cookies, localStorage, and sessionStorage — covers most auth implementations including JWT and session-based.
Q3. Is role-based testing possible when tests run in parallel? ▶

Yes — create one storageState file per role and assign them to separate projects in playwright.config.js.

// global-setup.js — login as each role
async function loginAs(role, email, password) {
  const browser = await chromium.launch();
  const page = await browser.newPage();
  await page.goto('/login');
  await page.fill('#email', email);
  await page.fill('#password', password);
  await page.click('button[type=submit]');
  await page.context().storageState({ path: `auth/${role}.json` });
  await browser.close();
}

export default async function globalSetup() {
  await loginAs('admin',  'admin@app.com',  'AdminPass1!');
  await loginAs('viewer', 'viewer@app.com', 'ViewPass1!');
  await loginAs('editor', 'editor@app.com', 'EditorPass1!');
}
// playwright.config.js — parallel role projects
export default {
  projects: [
    {
      name: 'admin-tests',
      use: { storageState: 'auth/admin.json' },
      testMatch: '**/admin/**/*.spec.js',
    },
    {
      name: 'viewer-tests',
      use: { storageState: 'auth/viewer.json' },
      testMatch: '**/viewer/**/*.spec.js',
    },
    {
      name: 'editor-tests',
      use: { storageState: 'auth/editor.json' },
      testMatch: '**/editor/**/*.spec.js',
    },
  ],
};
Each project worker runs in complete isolation — different cookies, different sessions, different roles, all in parallel.

Q5. What is the Page Object Model and why use it? ▶

POM encapsulates page locators and interactions into a class — tests call methods instead of writing raw selectors.

// pages/login.page.js
export class LoginPage {
  constructor(page) {
    this.page = page;
    this.emailInput    = page.locator('#email');
    this.passwordInput = page.locator('#password');
    this.submitBtn     = page.locator('button[type=submit]');
    this.errorMsg      = page.locator('.error-message');
  }

  async login(email, password) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.submitBtn.click();
    await this.page.waitForURL('**/dashboard');
  }

  async getError() {
    return this.errorMsg.textContent();
  }
}

// tests/login.spec.js
import { LoginPage } from '../pages/login.page.js';

test('valid login', async ({ page }) => {
  const login = new LoginPage(page);
  await login.login('user@app.com', 'pass123');
  await expect(page).toHaveURL(/dashboard/);
});
Benefits: Locator changes require 1 update (in page class) instead of N updates across tests. Tests read like plain English.
Q6. How do you manage test data in Playwright? ▶

Three patterns depending on complexity:

// 1. JSON fixtures (recommended for static data)
// test-data/users.json
{
  "admin": { "email": "admin@app.com", "password": "Admin1!" },
  "viewer": { "email": "viewer@app.com", "password": "View1!" }
}

// In test:
import users from '../test-data/users.json' assert { type: 'json' };
await login.login(users.admin.email, users.admin.password);
// 2. Playwright fixtures (for shared page/helper instances)
import { test as base } from '@playwright/test';
import { LoginPage } from './pages/login.page.js';

export const test = base.extend({
  loginPage: async ({ page }, use) => {
    await use(new LoginPage(page));
  },
  testUser: async ({}, use) => {
    await use({ email: 'user@app.com', password: 'pass' });
  },
});

// In test — no imports needed, fixtures injected
test('login', async ({ loginPage, testUser }) => {
  await loginPage.login(testUser.email, testUser.password);
});
// 3. API-created test data (freshest, no stale state)
const apiContext = await request.newContext({ baseURL: 'https://api.app.com' });
const user = await apiContext.post('/users', {
  data: { name: 'Test User', role: 'editor' }
});
const { id } = await user.json();
// cleanup after test
await apiContext.delete(`/users/${id}`);
Q7. What is a Playwright fixture and how is it different from a helper? ▶

Fixtures are Playwright's built-in DI system — they set up and tear down resources automatically per test.

// fixtures/index.js
import { test as base } from '@playwright/test';
import { LoginPage }     from '../pages/login.page.js';
import { DashboardPage } from '../pages/dashboard.page.js';

export const test = base.extend({
  // auto-login fixture — runs before each test, auto-tears down
  authenticatedPage: [async ({ page }, use) => {
    const login = new LoginPage(page);
    await login.login('user@app.com', 'pass');
    await use(page);          // test runs here
    await page.close();      // teardown after test
  }, { auto: false }],

  dashboard: async ({ page }, use) => {
    await use(new DashboardPage(page));
  },
});

// Usage — fixtures are injected, not imported
test('dashboard test', async ({ authenticatedPage, dashboard }) => {
  await dashboard.navigateTo();
});
FixtureHelper/Utility
LifecycleManaged by Playwright (setup + teardown)Manual — you call it
InjectionAuto-injected via destructuringImport and instantiate
Scopetest, worker, or projectNo scope concept
Best forPages, auth, DB connectionsOne-off utilities, formatters

Parallel tests must be independent — no shared state, no shared test data that gets mutated. Use unique test data per worker.
Q10. When do you configure explicit waits in Playwright? ▶

Playwright auto-waits for actionability (visible, enabled, stable, not obscured). Explicit waits are needed for:

ScenarioExplicit Wait
Wait for API call to completepage.waitForResponse()
Wait for navigationpage.waitForURL()
Wait for element statelocator.waitFor({ state })
Custom JS conditionpage.waitForFunction()
New page/popuppage.waitForEvent('popup')
Downloadpage.waitForEvent('download')
// Wait for API after clicking
const [response] = await Promise.all([
  page.waitForResponse(r => r.url().includes('/api/users') && r.status() === 200),
  page.click('#load-users'),
]);

// Wait for element visibility
await page.locator('.toast').waitFor({ state: 'visible', timeout: 5000 });

// Custom JS condition
await page.waitForFunction(() => window.appReady === true);

// Wait for new popup window
const [popup] = await Promise.all([
  page.waitForEvent('popup'),
  page.click('#open-popup'),
]);
Q11. How do you configure explicit waits in the Playwright config file? ▶ 
// playwright.config.js
export default {
  timeout: 60_000,            // per-test timeout (default: 30s)

  expect: {
    timeout: 5_000,           // expect() assertion timeout
  },

  use: {
    actionTimeout: 10_000,    // click, fill, select, etc.
    navigationTimeout: 30_000, // goto(), waitForURL()
  },
};
// Per-test override
test('slow operation', async ({ page }) => {
  test.setTimeout(120_000);
  await page.goto('/slow-page', { timeout: 60_000 });
  await expect(page.locator('h1')).toBeVisible({ timeout: 15_000 });
});
Set global timeouts conservatively. Override per-test for genuinely slow operations. Never use waitForTimeout() as a substitute for proper waits.
Q12. What are beforeAll, beforeEach, afterAll, afterEach and when to use each? ▶ 
test.describe('Product suite', () => {

  // Runs ONCE before all tests in this describe — use for expensive setup
  test.beforeAll(async ({ browser }) => {
    // create shared browser context, seed DB, start server
  });

  // Runs before EACH test — use for navigation, state reset
  test.beforeEach(async ({ page }) => {
    await page.goto('/products');
  });

  // Runs after EACH test — use for cleanup, screenshots on failure
  test.afterEach(async ({ page }, testInfo) => {
    if (testInfo.status !== testInfo.expectedStatus) {
      await page.screenshot({ path: `fail-${testInfo.title}.png` });
    }
  });

  // Runs ONCE after all tests — teardown, close connections
  test.afterAll(async () => {
    // delete test data, close DB, stop server
  });
});
HookFrequencyCommon Use
beforeAllOnce per describeCreate shared context, seed DB
beforeEachBefore every testNavigate, reset state, set auth
afterEachAfter every testScreenshot on failure, cleanup created records
afterAllOnce per describeClose connections, delete test data

Locators & Stability

Q14. What locator strategies does Playwright provide? ▶ 
// Role-based (most resilient — mirrors accessibility tree)
page.getByRole('button', { name: 'Submit' });
page.getByRole('textbox', { name: 'Email' });
page.getByRole('link',   { name: 'Home' });

// By label (for form inputs)
page.getByLabel('Email address');

// By placeholder
page.getByPlaceholder('Search...');

// By test ID (most stable — added by devs for testing)
page.getByTestId('submit-btn');  // data-testid="submit-btn"

// By text content
page.getByText('Welcome back');
page.getByText(/welcome/i);      // regex, case-insensitive

// CSS / XPath (last resort)
page.locator('#submit-btn');
page.locator('xpath=//button[@type="submit"]');
StrategyStabilityWhen to Use
getByRole⭐⭐⭐⭐⭐First choice — semantic, A11y aligned
getByTestId⭐⭐⭐⭐⭐When devs add data-testid attributes
getByLabel⭐⭐⭐⭐Form fields with labels
getByText⭐⭐⭐Text that won't change, buttons/links
CSS selector⭐⭐When above options fail
XPath⭐Last resort only
Q15. What do you do when a locator resolves to multiple elements? ▶ 
// BAD — throws if multiple matches found
await page.locator('.btn').click();

// GOOD — narrow by index
await page.locator('.btn').first().click();
await page.locator('.btn').last().click();
await page.locator('.btn').nth(2).click();   // 0-indexed

// BETTER — narrow the selector itself
await page.locator('form#checkout .btn-submit').click();

// BEST — scope to parent element
const form = page.locator('form#checkout');
await form.getByRole('button', { name: 'Submit' }).click();

// Count how many matches
const count = await page.locator('.item').count();
console.log(`Found ${count} items`);

// Iterate all matches
for (const item of await page.locator('.item').all()) {
  console.log(await item.textContent());
}

// Filter from multiple — find row with specific text
const row = page.locator('tr').filter({ hasText: 'John Doe' });
await row.getByRole('button', { name: 'Delete' }).click();
Q16. Would you use force: true while interacting with elements? Why or why not? ▶

Avoid it — force: true bypasses Playwright's actionability checks:

Check BypassedRisk
VisibilityClicking hidden elements — test passes but real user can't
Enabled stateInteracting with disabled inputs — masks bugs
StabilityClicking animating elements — race condition
Not obscuredClick goes to wrong element under overlay
// AVOID
await page.locator('#hidden-btn').click({ force: true });

// BETTER — wait for element to become interactable
await page.locator('#btn').waitFor({ state: 'visible' });
await page.locator('#btn').click();

// Close overlay first, then click
await page.locator('.modal-close').click();
await page.locator('#btn').click();

// Scroll into view first
await page.locator('#btn').scrollIntoViewIfNeeded();
await page.locator('#btn').click();
Acceptable uses: negative tests (verify disabled button ignores clicks), elements intentionally off-screen but functionally actionable (rare edge case).
Q17. How do you interact with iframes in Playwright? ▶ 
// Locate frame by URL
const frame = page.frameLocator('iframe[src*="payment"]');
await frame.getByLabel('Card Number').fill('4111111111111111');
await frame.getByRole('button', { name: 'Pay' }).click();

  .frameLocator('#outer-frame')
  .frameLocator('#inner-frame');

Q18. How do you handle shadow DOM elements? ▶ 
// Playwright CSS selectors pierce shadow DOM automatically with >>>

// Using locator chaining

Q19. How do you perform network mocking in Playwright? ▶ 
// Mock a GET response
await page.route('**/api/products', async route => {
  await route.fulfill({
    status: 200,
    contentType: 'application/json',
    body: JSON.stringify([{ id: 1, name: 'Mock Product', price: 99 }]),

// Abort request (simulate offline)
await page.route('**/api/analytics', route => route.abort());

// Modify request headers (inject auth token)
await page.route('**/api/**', async route => {
  await route.continue({
    headers: { ...route.request().headers(), 'Authorization': 'Bearer test-token' },
  });
});

// Assert request body (verify POST payload)
await page.route('**/api/order', async route => {
  const body = route.request().postDataJSON();
  expect(body.item).toBe('Widget');
  expect(body.quantity).toBe(3);
  await route.continue();
});

// Remove mock after use
await page.unroute('**/api/products');
Q20. How do you use Playwright's API testing (request context)? ▶ 
// playwright.config.js
use: { baseURL: 'https://api.app.com' }

// In test — use built-in 'request' fixture
test('create and verify user via API', async ({ request, page }) => {

  // CREATE via API (fast setup)
  const res = await request.post('/users', {
    data: { name: 'John', email: 'john@test.com', role: 'admin' },
  });
  expect(res.status()).toBe(201);
  const { id } = await res.json();

  // VERIFY in UI
  await page.goto(`/users/${id}`);
  await expect(page.getByText('John')).toBeVisible();

  // CLEANUP via API (fast teardown)
  await request.delete(`/users/${id}`);
});
API testing in Playwright is ideal for test setup/teardown — create data via API, verify in UI, clean up via API. Avoids slow UI flows for preconditions.

Q30. What is page.evaluate() and when do you use it? ▶ 
// Execute JS in browser context — returns result to Node

// Read localStorage
const token = await page.evaluate(() => localStorage.getItem('auth-token'));

// Set localStorage (bypass login for fast setup)
await page.evaluate(token => localStorage.setItem('auth-token', token), 'my-token');

// Scroll to bottom of page
await page.evaluate(() => window.scrollTo(0, document.body.scrollHeight));

// Read computed CSS property
const color = await page.evaluate(() => {
  const el = document.querySelector('#header');
  return window.getComputedStyle(el).getPropertyValue('background-color');
});

// Trigger custom events
await page.evaluate(() => {
  document.querySelector('#input').dispatchEvent(new Event('change'));
});

// evaluateHandle — returns ElementHandle (for DOM node)
const handle = await page.evaluateHandle(() => document.body);
Avoid overusing evaluate() — prefer Playwright's built-in locators. Use evaluate() when no native API covers the scenario (localStorage, custom events, computed styles).

Q31. Replace each character in a string with the next alphabetical character ▶ 
function shiftChars(str) {
  return str
    .split('')
    .map(char => {
      if (char === 'z') return 'a';          // wrap lowercase
      if (char === 'Z') return 'A';          // wrap uppercase
      if (/[a-zA-Z]/.test(char)) {
        return String.fromCharCode(char.charCodeAt(0) + 1);
      }
      return char;  // non-alpha unchanged
    })
    .join('');
}

console.log(shiftChars('Hello'));    // "Ifmmp"
console.log(shiftChars('xyz'));      // "yza"
console.log(shiftChars('aZ9!'));     // "bA9!"
console.log(shiftChars('Hello World 123!'));  // "Ifmmp Xpsme 123!"
Key insight: charCodeAt() + 1 shifts the character. Handle 'z'→'a' and 'Z'→'A' wrapping separately. Non-alphabetic characters pass through unchanged.
Q32. Fetch mobile price from two websites and return the cheapest ▶ 
import { chromium } from '@playwright/test';

async function getPrice(browser, siteName, url, selector) {
  const page = await browser.newPage();
  try {
    await page.goto(url, { waitUntil: 'domcontentloaded', timeout: 30_000 });
    const priceText = await page.locator(selector).first().textContent();
    const price = parseFloat(priceText.replace(/[^0-9.]/g, ''));
    console.log(`${siteName}: $${price}`);
    return { site: siteName, price };
  } catch (e) {
    console.error(`Failed to get price from ${siteName}:`, e.message);
    return { site: siteName, price: Infinity };
  } finally {
    await page.close();
  }
}

async function getCheapestPrice() {
  const browser = await chromium.launch();

  const sites = [
    { name: 'Amazon',  url: 'https://amazon.com/dp/PRODUCT_ID',   selector: '.a-price-whole' },
    { name: 'BestBuy', url: 'https://bestbuy.com/site/PRODUCT_ID', selector: '.priceView-customer-price span' },
  ];

  // Fetch both prices in parallel
  const results = await Promise.all(
    sites.map(s => getPrice(browser, s.name, s.url, s.selector))
  );

  await browser.close();

  const cheapest = results.reduce((min, curr) =>
    curr.price < min.price ? curr : min
  );

  console.log(`\n🏆 Cheapest: ${cheapest.site} at $${cheapest.price}`);
  return cheapest;
}

getCheapestPrice();
// Amazon: $1199.99
// BestBuy: $1249.99
// Cheapest: Amazon at $1199.99
Key: open pages in parallel with Promise.all(), handle errors gracefully (return Infinity on failure so they never win), always close pages in finally block.

Q34. How would you implement a retry wrapper for flaky Playwright actions? ▶ 
// utils/retry.js
export async function withRetry(fn, { retries = 3, delay = 1000, name = 'action' } = {}) {
  let lastError;
  for (let attempt = 1; attempt <= retries; attempt++) {
    try {
      return await fn();
    } catch (err) {
      lastError = err;
      console.warn(`[Retry ${attempt}/${retries}] ${name}: ${err.message}`);
      if (attempt < retries) await new Promise(r => setTimeout(r, delay));
    }
  }
  throw lastError;
}

// Usage
await withRetry(
  () => page.locator('.dynamic-element').click(),
  { retries: 3, delay: 500, name: 'click dynamic element' }
);

// expect.poll() — Playwright's built-in polling assertion
await expect.poll(async () => {
  return await page.locator('.status').textContent();
}, {
  message: 'Status should become Complete',
  timeout: 30_000,
  intervals: [1000, 2000, 5000]
}).toBe('Complete');
